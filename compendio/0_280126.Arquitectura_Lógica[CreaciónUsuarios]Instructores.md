# 1. CONTEXTO GENERAL DEL SISTEMA

El sistema corresponde a una **plataforma académica de simulación aeronáutica** orientada a:

-   Gestión de programas de entrenamiento
-   Asignación y ejecución de sesiones de simulador
-   Reserva de turnos (integración con SuperSaaS)
-   Administración de instructores y estudiantes
-   Control de progresión académica (PF / PM / OBS)

El sistema separa claramente:

-   **Dominio académico** (progreso, sesiones, fases)
-   **Dominio operativo/logístico** (reservas, turnos, calendarios externos)

----------

# 🧱 2. ESTRUCTURA ARQUITECTÓNICA COMPLETA

## 🔹 Jerarquía Académica

El sistema sigue una estructura jerárquica:

program  
 ↓  
program_version  
 ↓  
program_course  
 ↓  
phase_session_template  
 ↓  
course_session

----------

## 🔹 Descripción de cada nivel

### `program`

Define el programa académico general  
Ej: A320 Type Rating

----------

### `program_version`

Versiona el programa  
Permite cambios sin afectar estudiantes activos

----------

### `program_course`

Segmenta el programa en cursos o módulos

----------

### `phase_session_template`

Define la estructura de sesiones:

-   Secuencia (session_sequence)
-   Tipo (PF / PM / OBS)
-   Reglas académicas

----------

### `course_session`

Instancia real ejecutable de una sesión:

-   Fecha
-   Hora
-   Instructor
-   Estado
-   Relación con reserva

Es la **unidad académica operativa principal**

----------

# 🔁 4. FLUJO LÓGICO DE DATOS

## 🔹 Flujo principal académico

user_account (STUDENT)  
 ↓  
student  
 ↓  
reservation (logística)  
 ↓  
course_session (académico)  
 ↓  
student_session_attendance (resultado)  
 ↓  
motor de progresión

----------

## 🔹 Flujo de reserva

1.  Estudiante solicita turno
2.  Se crea `reservation`
3.  Se genera `course_session`
4.  Se vinculan ambas entidades

----------

## 🔹 Flujo de ejecución

1.  Sesión ocurre
2.  Instructor confirma asistencia
3.  Se actualiza `student_session_attendance`
4.  Se ejecuta motor de progresión

----------

## 🔹 Flujo de reprogramación

1.  Admin selecciona `course_session`
2.  Se obtiene `reservation`
3.  Se cancela booking externo
4.  Se genera nuevo booking
5.  Se actualiza:
    -   reservation
    -   course_session

----------

# ⚙️ 5. DECISIONES DE DISEÑO IMPORTANTES

## 🔹 Separación de dominios

Se definieron dos dominios independientes:

### Académico

-   course_session
-   program_course
-   phase_session_template
-   student_session_attendance

### Operativo

-   reservation
-   simulator_turn
-   integración SuperSaaS

----------

## 🔹 Identidad desacoplada

-   `user_account` no depende de student ni instructor
-   Permite múltiples perfiles futuros

----------

## 🔹 Relación 1:1 controlada

Entre:

reservation ↔ course_session

Se mantiene bidireccional para:

-   navegación eficiente
-   consistencia de dominio

----------

## 🔹 Asistencia desacoplada

La asistencia no vive en `course_session`

Se modela en:

student_session_attendance

Permitiendo:

-   múltiples participantes
-   evaluaciones individuales
-   escalabilidad futura

----------

## 🔹 Progresión basada en eventos

El avance académico no depende de la programación, sino de:

student_session_attendance

Esto evita inconsistencias por:

-   cancelaciones
-   reprogramaciones
-   no-shows

----------

## 🔹 Integración externa desacoplada

SuperSaaS se usa solo para:

-   booking
-   disponibilidad

No define el estado académico.

----------

# 📌 CONCLUSIÓN

El sistema presenta una arquitectura:

-   **Modular**
-   **Escalable**
-   **Orientada a dominio**
-   **Separada entre académico y operativo**

La entidad central del sistema es:

> `course_session` como núcleo académico  
> `student_session_attendance` como fuente de verdad de progresión
# 1. ENDPOINTS IMPLEMENTADOS Y FUNCIONALIDAD

## 🔹 Autenticación

### `POST /api/auth/login`

-   Autentica usuario (`user_account`)
-   Genera JWT con roles
-   Permite acceso condicionado por `ProtectedRoute` y `roleGuard`

----------

## 🔹 Administración de Usuarios

### `POST /api/admin/users`

-   Crea usuario en `user_account`
-   Asigna rol en `user_role`
-   Soporta múltiples roles (ADMIN, DIRECTOR, PROGRAMADOR, INSTRUCTOR, STUDENT)

----------

## 🔹 Administración de Instructores

### `POST /api/admin/instructors`

-   Valida que el usuario tenga rol INSTRUCTOR
-   Crea registro en `instructor`
-   Inicializa:
    -   instructor_type
    -   ranking
    -   status
    -   active

----------

### `GET /api/admin/instructors`

-   Lista instructores
-   Join con `user_account`

----------

### `PUT /api/admin/instructors/:id`

-   Actualiza:
    -   instructor_type
    -   ranking
    -   status
    -   active

----------

## 🔹 Reservas (Estudiante)

### `POST /api/reservations`

-   Crea registro en `reservation`
-   Integra con SuperSaaS (booking externo)
-   Almacena:
    -   simulator_turn_id
    -   fechas
    -   supersaas_booking_id

⚠️ Actualmente:

-   No crea `course_session`

----------

## 🔹 Administración de Reservas

### `GET /api/admin/reservations`

-   Obtiene reservas con información académica/logística

----------

### `GET /api/admin/reprogram/slots`

-   Obtiene disponibilidad futura para reprogramación
-   Usa:
    -   aircraft
    -   type (PF/PM)
    -   student_id

----------

### `POST /api/admin/reprogram`

-   Reprograma sesión existente
-   Flujo:
    1.  Bloquea reserva
    2.  Cancela booking en SuperSaaS
    3.  Genera nuevo booking
    4.  Actualiza `reservation`
    5.  Actualiza `course_session`

----------

### `PATCH /api/admin/reservations/:id/cancel`

-   Cancela reserva
-   Registra:
    -   cancelled_by
    -   cancellation_reason

----------

# ⚠️ 3. VALIDACIONES IMPORTANTES

## 🔹 Usuarios

-   Email único
-   Documento estructurado (pendiente mejora UI tipo documento)
-   Rol obligatorio

----------

## 🔹 Instructores

-   Usuario debe tener rol INSTRUCTOR
-   Validación de existencia de user_account
-   Edición controlada por admin

----------

## 🔹 Reservas

-   Validación de slot disponible
-   Integración correcta con SuperSaaS
-   Validación de schedule_id por aircraft/type

----------

## 🔹 Reprogramación

-   Uso de `FOR UPDATE` para evitar condiciones de carrera
-   Cancelación previa obligatoria en SuperSaaS
-   Transacción completa (BEGIN / COMMIT / ROLLBACK)

----------

## 🔹 Seguridad

-   JWT obligatorio
-   roleGuard por endpoint
-   Interceptor de Axios para envío de token

----------

# 🚨 4. PROBLEMAS DETECTADOS

## 🔴 1. Desacople crítico

Las reservas:

NO generan course_session

Impacto:

-   No existe sesión académica real
-   El dashboard académico queda incompleto
-   El sistema funciona solo a nivel logístico

----------

## 🔴 2. Flujo académico incompleto

Faltan:

-   Creación automática de `course_session`
-   Creación de `student_session_attendance`
-   Motor de progresión

----------

## 🔴 3. Doble relación no gestionada

reservation ↔ course_session

Existe estructuralmente pero no se mantiene en lógica.

----------

## 🔴 4. Progresión no implementada

No hay lógica para:

-   Descontar PF / OBS
-   Avanzar fases
-   Validar completitud

----------

## 🔴 5. Dependencia operativa excesiva

SuperSaaS controla:

-   disponibilidad
-   booking

Pero no debe controlar:

-   estado académico

----------
