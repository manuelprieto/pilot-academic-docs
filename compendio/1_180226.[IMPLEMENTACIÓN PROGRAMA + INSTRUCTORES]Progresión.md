# DOCUMENTO TÉCNICO – ARQUITECTURA DEL SISTEMA ACADÉMICO

----------

# 1️⃣ Contexto General del Sistema

El sistema corresponde a una **plataforma académica aeronáutica** orientada a la gestión integral de:

-   Programas de formación (ej. A320, Transición, Refresher)
-   Versiones de programas (intensivo, regular, etc.)
-   Ejecución operativa de sesiones (simulador, teoría)
-   Control de progreso académico por estudiante
-   Gestión de reservas y asistencia
-   Aplicación de reglas disciplinarias (infraestructura futura)

El sistema separa claramente:

-   **Estructura académica (currículo)**
-   **Ejecución operativa (reservas y sesiones)**
-   **Progreso académico (estado del estudiante)**

----------

# 2️⃣ Estructura Arquitectónica General

El sistema sigue una arquitectura jerárquica con versionamiento y desacoplamiento entre diseño académico y ejecución operativa.

## Jerarquía principal:

program  
 └── program_version  
 └── program_course  
 └── course_session

Paralelamente:

program_structure_version  
 └── phase_session_template

Y finalmente:

enrollment  
 └── student_phase_progress

----------

# 3️⃣ Modelo de Datos – Entidades Principales

----------

## 3.1 Programas Académicos

### program

Entidad raíz del sistema académico.

Define el producto formativo institucional.

**Responsabilidad:**

-   Identidad del programa
-   Prioridad institucional
-   Duración máxima

----------

### program_version

Permite múltiples configuraciones de un mismo programa.

**Responsabilidad:**

-   Variantes académicas (intensivo, regular, etc.)
-   Parámetros cuantitativos del programa

----------

### program_course

Capa operativa asociada a la ejecución del programa.

**Responsabilidad:**

-   Configuración de calendario
-   Modalidad de programación (ej. FLEXIBLE)
-   Capacidad operativa

**Nota:**  
No define estructura académica, sino condiciones de ejecución.

----------

## 3.2 Estructura Académica

### program_structure_version

Define una versión congelada de la estructura curricular.

**Responsabilidad:**

-   Garantizar trazabilidad histórica
-   Evitar que cambios afecten estudiantes activos

----------

### phase_session_template

Unidad mínima académica.

Representa cada sesión del currículo.

**Ejemplos:**

-   Clase 1 Tierra
-   PF Jet 1
-   PF Jet 2

**Responsabilidad:**

-   Definir contenido académico
-   Tipo de sesión (THEORY, PF, PM)
-   Orden lógico

----------

## 3.3 Ejecución Académica

### course_session

Representa una instancia real de una sesión académica.

**Relaciones:**

-   program_course (operación)
-   phase_session_template (currículo)
-   enrollment (contexto del estudiante)

**Responsabilidad:**

-   Materializar la sesión en tiempo real
-   Servir de puente entre currículo y operación

----------

### reservation

Entidad operativa que representa la reserva de un estudiante.

**Responsabilidad:**

-   Gestión de agenda
-   Registro de asistencia
-   Evento real del sistema

Contiene:

-   attendance_status
-   course_session_id
-   student_id

Es el disparador del motor académico.

----------

## 3.4 Matrícula y Progreso

### enrollment

Representa la vinculación del estudiante a un programa.

**Responsabilidad:**

-   Contexto académico del estudiante
-   Estado global (ACTIVE, BLOCKED, COMPLETED)

Incluye:

-   program_id
-   program_version_id
-   program_structure_version_id

----------

### student_phase_progress

Tabla central del progreso académico.

**Responsabilidad:**

-   Seguimiento por sesión
-   Control de estados

Estados:

-   LOCKED
-   PENDING
-   COMPLETED
-   FAILED

Constraint crítico:

UNIQUE (enrollment_id, phase_session_template_id)

Garantiza integridad del progreso.

----------

# 5️⃣ Flujo Lógico de Datos

----------

## 5.1 Inicialización (Matrícula)

1.  Se crea un enrollment
2.  Se identifica la estructura activa
3.  Se generan registros en student_phase_progress:
    -   Primera sesión → PENDING
    -   Resto → LOCKED

----------

## 5.2 Ejecución (Reserva)

1.  Se crea una reservation
2.  Se asocia a una course_session
3.  La course_session representa la sesión real del currículo

----------

## 5.3 Asistencia (Evento Crítico)

1.  Se actualiza attendance_status en reservation
2.  Este evento dispara el motor académico
3.  Se identifica la sesión curricular correspondiente

----------

## 5.4 Progreso

1.  Se actualiza student_phase_progress:
    -   COMPLETED o FAILED
2.  Se evalúa desbloqueo de siguiente sesión
3.  Se mantiene consistencia por enrollment

----------

# 6️⃣ Decisiones de Diseño Importantes

----------

## 6.1 Separación de responsabilidades

Capa

Función

Ejecución

Progreso

----------

## 6.2 Uso de versionamiento

-   program_structure_version congela currículo
-   enrollment referencia versión fija
-   Garantiza trazabilidad histórica

----------

## 6.3 Motor basado en eventos

El sistema es **event-driven**:

Evento real → reservation.attendance_status

No se basa en:

-   course_session
-   cron jobs

----------

## 6.4 Single Source of Truth

Dominio

Fuente

Asistencia

reservation

Progreso

student_phase_progress

Estado global

enrollment

----------

## 6.5 Desacoplamiento académico vs operativo

-   program_course → operación
-   phase_session_template → currículo

Evita mezcla de lógica.

----------

## 6.6 Preparación para escalabilidad

El modelo soporta:

-   Repetición de programas
-   Versionamiento histórico
-   Múltiples sesiones por estudiante
-   Diferenciación PF / PM
-   Extensión a infracciones

----------

-   Diseño curricular

-   Progreso del estudiante

Esto permite evolución controlada del sistema sin afectar datos históricos.

# 1️⃣ Endpoints creados y funcionalidad

----------

## 🔹 Matrícula

### `POST /api/enrollments`

**Funcionalidad:**

-   Crea un enrollment (matrícula)
-   Inicializa el progreso académico del estudiante

**Acciones internas:**

1.  Valida datos de entrada
2.  Inserta registro en `enrollment`
3.  Obtiene estructura activa (`program_structure_version`)
4.  Ejecuta initializer:
    -   Genera registros en `student_phase_progress`
    -   Primera sesión → `PENDING`
    -   Resto → `LOCKED`

----------

## 🔹 Gestión Administrativa de Reservas

### `GET /api/admin/reservations`

**Funcionalidad:**

-   Consulta reservas administrativas

----------

### `GET /api/admin/reprogram/slots`

**Funcionalidad:**

-   Obtiene disponibilidad de slots para reprogramación

**Validaciones:**

-   aircraft
-   type
-   from_date
-   course_session_id

----------

### `POST /api/admin/reprogram`

**Funcionalidad:**

-   Reprograma una sesión existente

**Acciones internas:**

1.  Bloquea reservation actual
2.  Cancela booking en sistema externo (SuperSaaS)
3.  Calcula nuevo turno
4.  Crea nuevo booking
5.  Actualiza:
    -   reservation
    -   course_session

----------

### `PATCH /api/admin/reservations/:id/attendance`

**Funcionalidad:**

-   Marca asistencia del estudiante
-   Dispara el motor académico

**Acciones internas:**

1.  Actualiza `reservation.attendance_status`
2.  Registra timestamp de instructor
3.  Ejecuta motor de progreso:
    -   Completa sesión
    -   Desbloquea siguiente
    -   (Futuro) crea infracción

----------

### `PATCH /api/admin/reservations/:id/cancel`

**Funcionalidad:**

-   Cancela una reserva

----------

# 3️⃣ Validaciones importantes

----------

## 🔹 En matrícula

-   `program_version` debe existir y estar activa
-   Debe existir `program_structure_version` activa
-   Integridad de datos antes de generar progreso

----------

## 🔹 En reservas

-   Validación de disponibilidad de slots
-   Validación de turno
-   Validación de schedule_id (SuperSaaS)
-   Bloqueo de registros en reprogramación

----------

## 🔹 En asistencia

-   `reservationId` obligatorio
-   `attendance_status` válido (CHECK constraint en DB)
-   Reserva debe existir

----------

## 🔹 En base de datos

-   `student_phase_progress`:
    
    UNIQUE (enrollment_id, phase_session_template_id)
    
-   `course_session`:
    -   `program_course_id` NOT NULL
-   `reservation`:
    -   integridad con course_session
    -   integridad con student

----------

# 4️⃣ Problemas detectados

----------

## 🔴 4.1 No automatización de course_session

-   Se crean manualmente
-   Rompe flujo natural
-   Genera inconsistencias

----------

## 🔴 4.2 Dependencia de `program_course_id`

-   `course_session` exige este campo
-   No pertenece al modelo académico nuevo
-   Genera fricción en pruebas

----------

## 🔴 4.3 Falta de `enrollment_id` en reservation

-   El motor depende de inferencias
-   Riesgo en escenarios de múltiples matrículas

----------

## 🔴 4.4 Estado híbrido del modelo

-   Mezcla modelo antiguo (program_course)
-   Con modelo nuevo (structure_version + template)

----------

## 🔴 4.5 Motor parcialmente integrado

-   Funciona conceptualmente
-   No validado completamente en flujo real
-   Falta integración total con reservas reales

----------

## 🔴 4.6 Falta de módulo de infracciones

-   NO_SHOW no genera consecuencias aún
-   No existe acumulación ni bloqueo

----------
