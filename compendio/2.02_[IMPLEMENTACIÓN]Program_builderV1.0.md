# DOCUMENTO DE ARQUITECTURA

## 🔷 Sistema Académico – Gestión de Programas, Matrícula y Progreso

----------

# 1️⃣ CONTEXTO GENERAL DEL SISTEMA

El sistema está diseñado para gestionar un modelo académico basado en:

Programas estructurados → versiones → módulos → fases → sesiones

Su objetivo principal es:

✔ Administrar la formación de estudiantes  
✔ Controlar la progresión académica por sesiones  
✔ Permitir seguimiento operativo (reservas, instructores)  
✔ Soportar lógica de avance secuencial

El dominio se comporta como un **LMS especializado (tipo simulación/instrucción técnica)** con restricciones de secuencia y estados.

----------

# 2️⃣ ESTRUCTURA ARQUITECTÓNICA

El sistema sigue una arquitectura por capas:

Frontend (React)  
 ↓  
API Layer (Express)  
 ↓  
Application Layer (Services / Business Logic)  
 ↓  
Persistence Layer (PostgreSQL)

----------

## 🔹 2.1 CAPAS

### 🟢 Presentación

-   Dashboard estudiante
-   Formularios administrativos
-   Visualización de progreso (timeline + tabla)

----------

### 🟡 Aplicación (Backend)

Responsable de:

✔ Orquestación de flujos (matrícula, progreso)  
✔ Validación de reglas de negocio  
✔ Inicialización de estructuras académicas

----------

### 🔵 Dominio (Lógica Académica)

Conceptos clave:

✔ Programas estructurados  
✔ Secuencia de aprendizaje  
✔ Progreso controlado por estados

----------

### ⚫ Persistencia

Base relacional (PostgreSQL) con entidades normalizadas.

----------

# 3️⃣ MODELO ACADÉMICO (CORE)

----------

## 🔷 3.1 JERARQUÍA PRINCIPAL

Program  
 ↓  
Program Structure Version  
 ↓  
Program Module  
 ↓  
Program Phase  
 ↓  
Phase Session Template

----------

## 🔹 DESCRIPCIÓN

### 📘 Program

Entidad raíz del sistema académico.

Representa un tipo de formación (ej: transición aerolínea)

----------

### 📗 Program Structure Version

Versiona la estructura académica  
Permite cambios sin afectar estudiantes existentes

----------

### 📙 Program Module

Agrupa fases por bloques lógicos  
Ej: Tierra, Simulador, Evaluación

----------

### 📕 Program Phase

Subdivisión pedagógica dentro de un módulo  
Define progresión conceptual

----------

### 📓 Phase Session Template

Define cada sesión académica  
Es el blueprint (plantilla)  
NO es ejecución real

Incluye:

✔ session_number  
✔ orden lógico  
✔ tipo de sesión

----------

# 4️⃣ MODELO OPERATIVO (EJECUCIÓN)

----------

## 🔷 4.1 ENTIDADES PRINCIPALES

User Account  
 ↓  
Student  
 ↓  
Enrollment  
 ↓  
Student Phase Progress

----------

## 🔹 DESCRIPCIÓN

### 👤 User Account

Entidad base de autenticación  
Puede tener múltiples roles

----------

### 🎓 Student

Extensión de usuario  
Incluye datos académicos

----------

### 📄 Enrollment

Relación entre:  
✔ estudiante  
✔ versión de programa

👉 Es el punto de inicio del flujo académico

----------

### 📊 Student Phase Progress

Entidad crítica del sistema  
Representa el estado REAL de cada sesión

Incluye:

✔ student_id  
✔ enrollment_id  
✔ phase_session_template_id  
✔ status  
✔ datos operativos (fecha, instructor, nota)

----------

# 5️⃣ RELACIONES ENTRE ENTIDADES

----------

## 🔷 5.1 RELACIÓN ACADÉMICA

program  
 → program_structure_version  
 → program_module  
 → program_phase  
 → phase_session_template

----------

## 🔷 5.2 RELACIÓN OPERATIVA

user_account  
 → student  
 → enrollment  
 → student_phase_progress  
 → phase_session_template

----------

## 🔷 5.3 RELACIÓN DE EJECUCIÓN

phase_session_template (plantilla)  
 ↓  
student_phase_progress (instancia real)

----------

# 6️⃣ FLUJO LÓGICO DE DATOS

----------

## 🔷 6.1 MATRÍCULA

1. Se crea enrollment  
2. Se selecciona program_structure_version  
3. Se dispara inicialización de progreso

----------

## 🔷 6.2 INICIALIZACIÓN

1. Se consultan sesiones (templates)  
2. Se ordenan por jerarquía académica  
3. Se crean registros en student_phase_progress  
 - primera sesión → PENDING  
 - resto → LOCKED

----------

## 🔷 6.3 PROGRESIÓN

Estado de sesión:  
LOCKED → PENDING → COMPLETED

Reglas implícitas:

✔ solo una sesión activa  
✔ secuencia estricta  
✔ desbloqueo progresivo

----------

## 🔷 6.4 VISUALIZACIÓN

Frontend consume:

student_phase_progress

Y construye:

✔ timeline  
✔ tabla  
✔ porcentaje de avance

----------

# 7️⃣ MODELO DE ESTADOS

----------

## 🔷 Estados definidos

LOCKED      → bloqueada  
PENDING     → disponible  
COMPLETED   → completada

----------

## 🔷 TRANSICIONES

LOCKED → PENDING  
PENDING → COMPLETED  
COMPLETED → (no reversible)

----------

# 8️⃣ DECISIONES DE DISEÑO IMPORTANTES

----------

## 🟢 1. SEPARACIÓN TEMPLATE vs EJECUCIÓN

phase_session_template ≠ student_phase_progress

✔ permite reutilización  
✔ permite versionamiento  
✔ evita duplicación lógica

----------

## 🟢 2. VERSIONAMIENTO DE PROGRAMAS

program_structure_version

✔ soporta evolución del currículo  
✔ evita romper datos históricos

----------

## 🟢 3. PROGRESO COMO ENTIDAD EXPLÍCITA

student_phase_progress

✔ evita cálculos dinámicos complejos  
✔ permite auditoría  
✔ permite tracking detallado

----------

## 🟢 4. INICIALIZACIÓN CONTROLADA

progreso generado al matricular

✔ consistencia  
✔ estructura fija  
✔ evita inconsistencias posteriores

----------

## 🟡 5. LÓGICA DISTRIBUIDA (ACTUAL)

Parte en backend  
Parte en frontend

👉 Esto es temporal (estado demo)

----------

## 🔴 6. CONTROL DE SECUENCIA (INCOMPLETO)

Actualmente parcialmente en frontend

👉 debe migrarse completamente al backend

----------

# RESUMEN TÉCNICO – LÓGICA DEL SISTEMA

## 🔷 Sistema Académico (Estado Demo)

----------

# 1️⃣ ENDPOINTS IMPLEMENTADOS

----------

## 🔹 1.1 Estudiantes

### `POST /api/admin/students`

**Funcionalidad:**

✔ Crea usuario base (user_account)  
✔ Asigna rol STUDENT  
✔ Crea entidad student  
✔ Vincula grupo académico (programming_group)  
✔ Inicializa PF (simulaciones disponibles)

----------

### `GET /api/admin/students`

**Funcionalidad:**

✔ Lista estudiantes activos  
✔ Usado para selección en matrícula (UI)

----------

## 🔹 1.2 Matrícula (Enrollment)

### `POST /api/admin/enrollments`

**Funcionalidad:**

✔ Crea enrollment (student + program_structure_version)  
✔ Dispara inicialización de progreso  
✔ Define punto de partida académico

----------

## 🔹 1.3 Progreso

### `GET /api/student/progress`

**Funcionalidad:**

✔ Devuelve progreso completo del estudiante  
✔ Incluye:  
 - estado de cada sesión  
 - fechas  
 - instructor  
 - notas  
✔ Fuente principal del dashboard

----------

## 🔹 1.4 Reservas

### `GET /api/reservations/history`

**Funcionalidad:**

✔ Retorna historial de reservas  
✔ Usado en dashboard del estudiante

----------

## 🔴 1.5 Completar Sesión (DEMO)

### `POST /api/admin/instructors/complete-session`

**Funcionalidad:**

✔ Marca sesión actual como COMPLETED  
✔ Busca siguiente sesión por orden  
✔ Cambia siguiente sesión a PENDING  
✔ Permite simular avance académico

----------

# 2️⃣ FLUJO DE NEGOCIO

----------

## 🔷 2.1 Autenticación

✔ Basado en JWT  
✔ authMiddleware protege rutas  
✔ roleGuard controla acceso por rol

⚠️ En demo:

❌ seguridad deshabilitada parcialmente  
❌ endpoints accesibles sin validación estricta

----------

## 🔷 2.2 Creación de Estudiante

1. Admin crea usuario  
2. Se asigna rol STUDENT  
3. Se crea entidad student  
4. Queda disponible para matrícula

----------

## 🔷 2.3 Matrícula

1. Se selecciona estudiante  
2. Se selecciona program_structure_version  
3. Se crea enrollment  
4. Se inicializa progreso automáticamente

----------

## 🔷 2.4 Inicialización de Progreso

1. Se consultan sesiones (phase_session_template)  
2. Se ordenan (intento jerárquico)  
3. Se crean registros en student_phase_progress:  
 - primera → PENDING  
 - resto → LOCKED

----------

## 🔷 2.5 Visualización (Dashboard)

1. Frontend consulta:  
 - /student/me  
 - /student/progress  
 - /reservations/history  
  
2. Construye:  
 ✔ barra de progreso (%)  
 ✔ timeline (sesiones)  
 ✔ tabla detallada

----------

## 🔷 2.6 Progresión (DEMO)

1. Usuario presiona "Completar"  
2. Se ejecuta:  
 - actualización local (frontend)  
 o  
 - endpoint demo (backend)  
  
3. Resultado:  
 ✔ sesión actual → COMPLETED  
 ✔ siguiente → PENDING

----------

----------

✔ inicia en última sesión

❌ ORDER BY incompleto o inconsistente

----------

✔ genera progreso  
  
Frontend:  
✔ decide estado (index === 0)

👉 riesgo de desincronización

----------

## 🔴 4.3 Uso de lógica hardcodeada

✔ index === 0 para sesión activa  
✔ labels definidos en frontend

----------

## 🔴 4.4 Endpoint de progreso no robusto

✔ /complete-session no valida reglas  
✔ permite ejecución arbitraria

----------

## 🔴 4.5 Seguridad relajada

✔ roleGuard removido en pruebas  
✔ authMiddleware ignorado en algunos casos

----------

## 🔴 4.6 Dependencia del frontend

✔ qué sesión es disponible  
✔ qué botón aparece
