# Arquitectura del Sistema Académico de Simulación

## 1. Contexto General del Sistema

El sistema corresponde a una plataforma académica para la gestión de entrenamiento en simuladores de aviación.  
Su objetivo es coordinar:

-   Programación académica estructurada
-   Reservas operativas (integradas con un sistema externo de calendarios)
-   Consolidación de sesiones académicas reales
-   Gestión administrativa por actores (PF, PM, Instructor)

El diseño separa explícitamente:

-   **Reserva operativa (booking)** → nivel transaccional
-   **Sesión académica (course_session)** → nivel pedagógico

Esto permite desacoplar la logística del entrenamiento de la estructura académica.

----------

## 2. Estructura Arquitectónica

El sistema sigue una arquitectura jerárquica basada en programas académicos:

program  
 └── program_version  
 └── program_course  
 └── course_session  
 └── reservation (PF / PM)

### Descripción de niveles

#### 🔹 program

Entidad raíz que define un programa académico (ej: habilitación B1900).

#### 🔹 program_version

Versionamiento del programa (permite evolución curricular sin afectar históricos).

#### 🔹 program_course

Representa la inscripción de un estudiante en un programa específico.

-   Relaciona estudiante + versión del programa
-   Es la unidad académica activa del estudiante

#### 🔹 course_session

Unidad académica ejecutable:

-   Define:
    -   fecha
    -   horario
    -   simulador
    -   instructor
    -   estado académico
-   Representa **una sesión real de entrenamiento**

#### 🔹 reservation

Participación individual dentro de una sesión:

-   PF (Pilot Flying)
-   PM (Pilot Monitoring)

No representa la sesión, sino la **ocupación de un slot por un actor**.

----------

## 3. Modelo de Datos y Relaciones

### 3.1 Entidades Principales

#### student

-   Representa al alumno
-   Contiene estado académico y métricas (ej: PF restantes)

#### user_account

-   Identidad del sistema (login)
-   Relacionado con student e instructor

#### instructor

-   Actor académico
-   Relacionado a user_account

----------

### 3.2 Estructura Académica

#### program

-   Configuración base del programa

#### program_version

-   Versionamiento del contenido académico

#### program_course

-   Relación estudiante ↔ programa
-   Punto de anclaje académico

----------

### 3.3 Ejecución Académica

#### course_session

Campos clave:

-   program_course_id
-   phase_session_template_id
-   session_sequence
-   session_date
-   start_time
-   end_time
-   instructor_id
-   status

Responsabilidad:

-   Representar la sesión académica real

----------

#### reservation

Campos clave:

-   student_id
-   simulator_unit_id
-   reservation_date
-   start_time
-   end_time
-   supersaas_booking_id
-   course_session_id (FK)
-   attendance_status
-   reprogram_status
-   cancelled_at

Responsabilidad:

-   Representar la participación individual

----------

### 3.4 Recursos

#### simulator_unit

-   Representa simuladores físicos (SIM1, SIM2)

#### simulator_turn

-   Define bloques de tiempo disponibles

#### calendar

-   Configuración de calendarios externos (roles PF / PM)

----------

## 4. Relaciones Clave

### 4.1 Relación central

course_session (1) ←── (N) reservation

-   Una sesión puede tener múltiples participantes
-   Cada reservation pertenece a una sola sesión (opcionalmente)

----------

### 4.2 Relación estudiante

student (1) ─── (N) reservation

----------

### 4.3 Relación académica

program_course (1) ─── (N) course_session

----------

### 4.4 Relación instructor

course_session (N) ─── (1) instructor

----------

## 5. Flujo Lógico de Datos

### 5.1 Flujo de Reserva (Operativo)

1.  Estudiante solicita disponibilidad
2.  Sistema evalúa reglas académicas
3.  Se valida disponibilidad externa (calendario)
4.  Se crea reservation
5.  Se descuenta capacidad académica (PF)

Resultado:

-   Existe participación, pero aún no necesariamente sesión consolidada

----------

### 5.2 Creación de Sesión Académica

Se activa cuando:

-   Existe al menos una reservation válida (normalmente PF)
-   Se tiene definido:
    -   fecha
    -   horario
    -   simulador

Proceso:

-   Se crea `course_session`
-   Se vinculan reservations mediante `course_session_id`

----------

### 5.3 Consolidación Académica

El sistema agrupa:

-   PF
-   PM
-   Instructor
-   Simulador

En una única `course_session`.

----------

### 5.4 Gestión Administrativa

Sobre `course_session` se gestionan:

-   estados académicos
-   asignaciones
-   reprogramaciones

Sobre `reservation` se gestionan:

-   asistencia (no_show)
-   cancelaciones individuales
-   reprogramación por actor

----------

## 6. Decisiones de Diseño Importantes

### 6.1 Separación de responsabilidades

Entidad

Responsabilidad

reservation

participación individual

course_session

sesión académica

----------

### 6.2 Desacoplamiento de sistema externo

-   `supersaas_booking_id` se usa solo como referencia
-   No es la fuente de verdad del modelo académico

----------

### 6.3 Normalización de actores

PF y PM:

-   No son columnas
-   Son instancias de reservation

----------

### 6.4 Estados intermedios

Se introducen estados como:

-   awaiting_pf_reprogram
-   awaiting_instructor_assignment

Permiten:

-   trazabilidad
-   evitar pérdida de información
-   flujos no destructivos

----------

### 6.5 Relación invertida correcta

Se adopta:

reservation → course_session

y no:

course_session → reservation

Esto permite:

-   múltiples participantes
-   reprogramación sin pérdida de datos
-   flexibilidad académica

----------

### 6.6 Modelo orientado a dominio

El sistema distingue claramente:

-   evento académico (session)
-   participación individual (reservation)

Esto evita:

-   duplicación de sesiones
-   inconsistencias de estado
-   pérdida de trazabilidad

----------

# Lógica del Sistema Académico de Simulación

## 1. Endpoints Creados y Funcionalidad

### 🔹 Módulo Estudiante (Reservas)

#### `GET /api/reservations/suggestions`

Genera sugerencias de slots disponibles.

Funcionalidad:

-   Aplica reglas académicas
-   Filtra por ventana horaria del estudiante
-   Evalúa disponibilidad real en sistema externo
-   Soporta concurrencia por simulador físico
-   Devuelve múltiples opciones priorizadas

----------

#### `POST /api/reservations`

Crea una reserva real.

Funcionalidad:

-   Bloquea estudiante (control de concurrencia)
-   Valida disponibilidad académica
-   Revalida disponibilidad externa
-   Crea booking en sistema externo
-   Persiste reservation local
-   Descuenta PF si aplica

----------

#### `GET /api/reservations/history`

Consulta historial del estudiante.

----------

----------

### 🔹 Módulo Administrativo (Sesiones)

#### `GET /api/admin/sessions`

Obtiene sesiones académicas consolidadas.

Funcionalidad:

-   Agrupa información de múltiples tablas
-   Consolida PF, PM, instructor y simulador
-   Retorna una entidad por sesión académica
-   Evita duplicidad mediante lógica de agrupación

----------

#### `PATCH /api/admin/sessions/:id`

Ejecuta acciones administrativas por actor.

Funcionalidad:

**Reprogramación**

-   PF → estado intermedio
-   PM → desvinculación controlada
-   Instructor → limpieza de asignación

**Cancelación**

-   PF → cancela participación
-   Instructor → libera sesión
-   Sesión → cancelación total

**Otros**

-   No Show → marca asistencia individual
-   Vista → trazabilidad

----------

## 2. Flujo de Negocio

----------

### 2.1 Autenticación

-   Usuario se autentica mediante `user_account`
-   Se determina rol:
    -   STUDENT
    -   ADMIN

----------

### 2.2 Enrolamiento

-   Se crea:
    -   `user_account`
    -   `student`
    -   `program_course`
-   Se asigna:
    -   grupo académico
    -   PF disponibles

❗ No se generan reservas ni sesiones en este punto

----------

### 2.3 Flujo de Reserva (Estudiante)

1.  Estudiante solicita sugerencias
2.  Sistema:
    -   valida reglas
    -   consulta disponibilidad externa
3.  Estudiante selecciona slot
4.  Se ejecuta reserva:
    -   creación externa
    -   persistencia local (`reservation`)
    -   descuento PF

Resultado:

-   Participación creada
-   Aún no necesariamente sesión académica

----------

### 2.4 Consolidación de Sesión

Se produce cuando:

-   Existe una reserva válida (normalmente PF)
-   Se tiene slot definido

Proceso:

-   Se crea `course_session` (manual o futuro automático)
-   Se vincula reservation mediante `course_session_id`

----------

### 2.5 Flujo Administrativo

1.  Admin consulta sesiones
2.  Visualiza:
    -   PF
    -   PM
    -   estado
3.  Ejecuta acciones:
    -   reprogramar
    -   cancelar
    -   marcar no_show
    -   asignar instructor

----------

### 2.6 Reprogramación

-   No elimina la sesión
-   No elimina inmediatamente la reserva
-   Marca estado intermedio

Permite:

-   trazabilidad
-   reasignación futura

----------

### 2.7 Asistencia (No Show)

-   Se gestiona a nivel individual (`reservation`)
-   No afecta directamente la existencia de la sesión

----------

## 3. Validaciones Importantes

----------

### 🔹 Académicas

-   PF disponibles (`pf_remaining`)
-   Límite diario de reservas
-   Reglas por estudiante (override posible)

----------

### 🔹 Temporales

-   Ventana horaria permitida por grupo
-   Fecha mínima de programación

----------

### 🔹 Disponibilidad

-   Validación contra sistema externo
-   Revalidación antes de confirmar reserva

----------

### 🔹 Concurrencia

-   Uso de `FOR UPDATE` sobre estudiante
-   Prevención de doble reserva

----------

### 🔹 Integridad

-   Validación de existencia de:
    -   turno
    -   calendario
    -   simulador

----------

## 4. Problemas Detectados

----------

### 🔴 1. Duplicación de sesiones en frontend

Causa:

-   Query devolvía múltiples filas por joins

Solución:

-   Agrupación correcta (una sesión = una fila)

----------

### 🔴 2. Uso incorrecto de `course_session.reservation_id`

Problema:

-   Relación 1:1 incorrecta
-   Pérdida de flexibilidad

Estado:

-   En transición hacia `reservation.course_session_id`

----------

### 🔴 3. Dependencia excesiva de `supersaas_booking_id`

Problema:

-   PF y PM pueden venir de calendarios distintos

Solución:

-   Uso de simulador + horario + sesión

----------

### 🔴 4. Eliminación destructiva en reprogramación

Problema:

-   Se perdía información del estudiante

Solución:

-   Estados intermedios (no destructivo)

----------

### 🔴 5. Inconsistencias visuales (frontend)

Problema:

-   PF desaparecía en estado intermedio

Solución:

-   Separar:
    -   nombre
    -   estado

----------

### 🔴 6. Falta de creación automática de sesiones

Problema:

-   Dependencia manual
